## Платформа для бронирования отелей
---
### Функциональные требования
- **Пользователь** может зарегестрироваться и создать профиль.
- **Пользователь** может искать отели по различным критериям (место, цена, дата).
- **Пользователь** может просматривать подробную информацию о каждом отеле (фотографии, описание, рейтинг, удобства).
- **Пользователь** может забронировать номер в отеле, выбрав дату заезда и выезда.
Пользователь может оплатить бронирование через систему.
- **Пользователь** может оставлять отзывы о пребывании в отеле.
- **Владелец** отеля может создать, редактировать и удалять объявления о своих отелях.
- **Владелец** отеля может управлять бронированиями и доступностью номеров.
- **Администратор** может управлять пользователями, отелями, бронированиями и отзывами.
---
### Бизнес-правила
- **Цена** за ночь не может быть отрицательной.
- **Отзыв** о номере можно оставить только после завершения бронирования.
- **Пользователь** может забронировать несколько номеров на одну дату, но только в пределах доступности отеля.
- **Оплата** бронирования должна быть завершена до начала периода проживания.
- **Отель** может иметь несколько типов номеров (например, стандартный, люкс).
- **Номер** может быть занят, забронирован или свободен, и его статус должен обновляться при изменении бронирования.

### Отношения
---
- Пользователь может сделать несколько бронирований — один ко многим.
- Отель может иметь несколько номеров — один ко многим.
- Номер может быть связан с несколькими типами номеров — многие ко многим (через тип номера).
- Отель может иметь несколько удобств — многие ко многим (через таблицу hotel_amenities).
- Пользователь может оставить несколько отзывов об отелях — один ко многим.
- Бронирование может включать несколько номеров — многие ко многим через таблицу booking_rooms.
- Бронирование связано с оплатой — один ко многим.
- Оплата может иметь несколько транзакций — один ко многим.
- Отель может иметь несколько фотографий — один ко многим.
- Номер может иметь несколько удобств — многие ко многим через таблицу room_amenities.
- Отель может иметь несколько историй изменения цен — один ко многим.
- Отель и бронирование могут быть связаны в таблице hotel_bookings — многие ко многим.
- Пользователь может иметь несколько предпочтений (тип номера, удобства) — многие ко многим (через таблицу user_preferences).
- Бронирование может иметь несколько изменений статуса — один ко многим (через таблицу booking_history).
- Отель может быть связан с несколькими владельцами — многие ко многим (через таблицу owners).
